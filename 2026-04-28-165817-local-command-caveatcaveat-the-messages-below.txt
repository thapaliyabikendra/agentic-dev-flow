  Bank Settings вҖ” Feature Specification

  Feature Overview

  Purpose

  Bank Settings provides bank administrators with centralized management of the
  LC Issuance Checklist вҖ” the standard verification items required during LC
  review workflows. Administrators can configure which checks appear, in what
  sequence, and which are mandatory for review completion. See
  http://localhost:8080/root/trade-finance/-/issues/15#1-purpose,
  http://localhost:8080/root/trade-finance/-/issues/16#1-purpose, and
  http://localhost:8080/root/trade-finance/-/issues/20#1-purpose.

  Scope

  In scope: View all configured checklist items in sequence, add new items with
  description, edit item descriptions, toggle items between active/inactive
  status, reorder items, and delete obsolete items. All operations preserve
  sequence numbering automatically. Out of scope: soft-delete (archive) strategy
   вҖ” currently hard-delete only; bulk operations; import/export of checklist
  templates; collaborative editing; version history; SLA metrics on checklist
  review time.

  Business Impact

  Eliminates manual checklist configuration in spreadsheets, standardizes LC
  review procedures across the bank, and provides an audit trail for regulatory
  compliance. Inactive items hide from front-line staff during LC review,
  reducing checklist cognitive load while preserving configuration history for
  re-activation.

  ---
  Open Blockers

  1. Soft-Delete vs Permanent Deletion Strategy (severity: high)

  1. FRS-BS-06 states "permanently remove obsolete or redundant checklist
  items," but open question #1 asks whether soft-delete (archive) should be used
   instead. The two approaches have different DDD implications: hard-delete
  simplifies the domain model but prevents recovery; soft-delete preserves audit
   history but requires filtering in all queries. This must be resolved before
  implementation begins, as it affects the Entity lifecycle, Command semantics,
  and Query filtering logic.

  1. Resolution needed: Should deleted items be hard-deleted (permanent removal)
   or soft-deleted (archived, hidden from queries)?

  Domain Layer Design

  Aggregates

  - http://localhost:8080/root/trade-finance/-/wikis/feats/bank-settings/entitie
  s/LCIssuanceChecklistItem вҖ” aggregate root for the checklist item lifecycle;
  uniquely identifies a verification requirement within a tenant's LC review
  workflow

  Value Objects

  - http://localhost:8080/root/trade-finance/-/wikis/feats/bank-settings/value-o
  bjects/ChecklistItemStatus вҖ” Active or Inactive enum; determines visibility to
   Branch and CTF staff during LC review
  - http://localhost:8080/root/trade-finance/-/wikis/feats/bank-settings/value-o
  bjects/SequenceNumber вҖ” immutable 1-based integer; recalculated automatically
  on reorder or delete to maintain deterministic ordering
  - http://localhost:8080/root/trade-finance/-/wikis/feats/bank-settings/value-o
  bjects/Description вҖ” immutable string, max 500 characters; truncated in table
  view with ellipsis if longer

  States

  - http://localhost:8080/root/trade-finance/-/wikis/feats/bank-settings/states/
  ChecklistItemLifecycleState вҖ” two-state machine: Active вҶ’ Inactive, Inactive вҶ’
   Active (toggle-only)

  Domain invariants

  - Inactive checklist items SHALL NOT appear in LC review checklists presented
  to Branch or CTF staff.
  - Checklist items SHALL always be displayed in configured sequence order (1,
  2, 3, ...) to maintain consistency.
  - Each item SHALL receive a unique sequence number within its tenant to
  maintain deterministic ordering.
  - Description length is capped at 500 characters.

  Error Handling, Auditing, and Logging

  - Domain rule violations (e.g., "Description is empty") throw
  BusinessException subclasses; ABP translates to 4xx ProblemDetails
  automatically.
  - Validation failures from FluentValidation surface as 400 Bad Request with
  field-level error details.
  - LCIssuanceChecklistItem uses FullAuditedAggregateRoot<Guid> вҖ” creation and
  modification audit fields (CreatedBy, CreatedOn, LastModifiedBy,
  LastModifiedOn) populated automatically.
  - Domain events are recorded in AbpAuditLog via ABP's audit event integration.
  - Structured logging via Serilog with TenantId and UserId enriched context;
  all command and query entry/exit points logged at Debug level.

  ---
  Performance and Scalability

  - Checklist item list query expected to serve 10 concurrent bank admins per
  tenant; composite index (TenantId, Status, SequenceNumber) supports this
  without full table scan.
  - LC review query (active items only) executes frequently during document
  processing; cached query result is invalidated on any checklist change (via
  domain event).
  - No N+1 risks identified вҖ” all queries use eager projections via Mapperly.
  - Sequence recalculation on reorder or delete is O(N) within tenant scope (N =
   item count, typically <В 200); acceptable for this low-frequency operation.

  - Schema change: AppChecklistItems table with indexes on (TenantId, Status,
  SequenceNumber) and (TenantId, CreatedOn). Standard DbMigrator run covers the
  migration.
  - No feature flags required; capability is gated by Role assignment (BankAdmin
   role).
  - No message queue bindings or external service calls required for M1.
