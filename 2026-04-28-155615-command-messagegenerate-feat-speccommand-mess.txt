  Bank Settings вҖ” Checklist Management вҖ” Feature Specification

  Feature Overview

  Purpose

  This feature delivers the administrative backend for the LC Issuance Checklist
   consumed by Branch and CTF reviewers during Letter of Credit review.
  BranchAdmin and CTFAdmin access the Private API surface to perform full CRUD
  over a per-tenant ordered list of ChecklistItem aggregates: list, get for
  pre-population, create, update, delete, toggle active/inactive status, and
  reorder one position up or down. See
  http://localhost:8080/root/trade-finance/-/issues/27#1-overview and
  http://localhost:8080/root/trade-finance/-/issues/28#6-business-rules.

  Scope

  In scope: per-tenant ChecklistItem aggregate with Description, IsActive, and
  Order properties; seven Private API endpoints; server-side FluentValidation
  matching FRS-BSC-07 rules; ABP audit logging; EF Core migration with
  AppChecklistItems table; Mapperly DTO projection where Sn = Order. Out of
  scope: search/filter endpoints, pagination, soft-delete recovery, bulk reorder
   (drag-and-drop), export, and description-uniqueness enforcement вҖ” all
  deferred to Phase 2/3 per FRS.

  Business Impact

  Provides Branch and CTF admins a durable, per-tenant configurable checklist
  that persists across sessions and is backed by a full audit trail, replacing
  any ephemeral client-side state with a regulated backend store.

  Gap analysis

  All Phase 1 UI flows are covered by the seven endpoints. No missing queries or
   commands for this milestone. The subscribeChecklist() reactive hook described
   in FRS-BSC-01 is client-side state management; the backend has no
  SignalR/push obligation in Phase 1. Audit readout (FRS Section 16) is
  satisfied by ABP's built-in audit log вҖ” no bespoke query needed.

  Error Handling, Auditing, and Logging

  - Domain rule violations throw BusinessException subclasses
  (BankSettings:ChecklistItem:DescriptionRequired,
  BankSettings:ChecklistItem:CannotMoveBeyondBounds); ABP translates to 4xx
  ProblemDetails.
  - FluentValidation failures surface as 400 with field-level details.
  - ChecklistItem uses FullAuditedAggregateRoot<Guid> вҖ” creation, modification,
  and soft-deletion audit fields populated automatically; satisfies all FRS
  Section 16 audit clauses.
  - AbpEntityChanges and AbpEntityPropertyChanges capture old/new values for
  description updates, status toggles, and order swaps.
  - Structured logging via Serilog with TenantId and UserId enriched on every
  request.

  Performance and Scalability

  - Admin lists are small (no pagination in PhaseВ 1); the composite index
  (TenantId, IsActive, Order) keeps list queries sub-linear per tenant.
  - Reorder writes touch exactly 2 rows (a swap); delete renumbers at most N
  rows; both are bounded by admin-scale list sizes.
  - The unique index (TenantId, Order) enforces the invariant at the DB level
  without application-level locks.
  - Phase 2 bulk reorder (drag-and-drop) will require a ReorderAllAsync in
  ChecklistItemManager вҖ” the architecture supports it without schema changes.

  Deployment Considerations

  - Schema change: AppChecklistItems table with two indexes. Run DbMigrator
  (Add_ChecklistItem migration) before deploying the new API host.
  - No feature flags; capability is gated by role assignment to
  TradeFinancePermissions.BankSettings.Default and the granular sub-permissions.
  - No RabbitMQ bindings required вҖ” no distributed events in this milestone.
  - Backward compatible вҖ” new table only; no existing table changes.

  Open Questions and Future Enhancements

  - Description uniqueness вҖ” FRS-BSC-02 explicitly allows duplicate descriptions
   in Phase 1; Phase 2 may add a per-tenant uniqueness check. The (TenantId, 
  Order) unique index does not constrain descriptions.
  - Pagination вҖ” FRS-BSC-01 defers pagination to PhaseВ 2. The ListChecklistItems
   query already accepts SkipCount/MaxResultCount via
  PagedAndSortedResultRequestDto; enabling it requires only a limit change.
  - Search/filter вҖ” no filter fields on ListChecklistItems in Phase 1; Phase 2
  should add DescriptionContains filter.
  - Confirmation dialog on delete вҖ” FRS-BSC-04 defers to PhaseВ 2. Backend is
  already hard-delete (soft-delete via ABP), so this is purely a UI change.
  - Bulk reorder / drag-and-drop вҖ” extend ChecklistItemManager with
  ReorderAllAsync(orderedIds) in Phase 2; no schema change required.
  - SignalR push вҖ” replace client-polling pattern with server-push for real-time
   admin sync across concurrent sessions; out of scope PhaseВ 1.
