# DDD Bridge Skill

Reads aggregated feature specs (from the `ddd-aggregate` skill) and creates GitLab milestones/issues and/or GSD phase artifacts. This is the delivery step — it turns documentation into actionable work items.

This skill does NOT generate DDD docs (use `ddd-docs`) or aggregate them (use `ddd-aggregate`). It CONSUMES aggregated specs and pushes them to project management tools.

## Prerequisites

Before using this skill, the feature must have:
- An aggregated spec in `docs/features/{feature-slug}/` containing at least `FEATURE_SPEC.md` (technical) or `ISSUE_SPEC.md` (business)
- If no spec exists, recommend running `ddd-aggregate` first

## Workflow

### Step 1 — Read Aggregated Spec

Locate the feature spec. The user may provide:
- A path: `docs/features/issue-lc/FEATURE_SPEC.md`
- A feature name: "Issue LC" → look for `docs/features/issue-lc/`
- A BC name: "lc-management" → look for `docs/features/lc-management/`

Read both `FEATURE_SPEC.md` and `ISSUE_SPEC.md` if they exist. The technical spec provides structured epic/story data; the business spec provides issue body content.

If neither exists, inform the user and suggest: "Run `/ddd-aggregate` first to generate the feature spec."

### Step 2 — Determine Target

Ask the user which targets to create:

| Target | What Gets Created | Source |
|--------|------------------|--------|
| **GitLab** | Milestone + epic issues + story issues + labels | FEATURE_SPEC.md Section 8 (epics/stories) + ISSUE_SPEC.md (body content) |
| **GSD** | REQUIREMENTS.md + phase stubs in .planning/ | FEATURE_SPEC.md Section 8 (epics → phases) + Section 3 (aggregates → CONTEXT.md) |
| **Both** | All of the above | Both spec files |

### Step 3 — GitLab Delivery

If GitLab is selected, follow this sequence:

**3a. Create Labels (if missing)**

Read `references/label-taxonomy.md` for the label naming conventions. Check which labels already exist using `mcp__gitlab__list_labels`. Only create missing ones using `mcp__gitlab__create_label`.

Labels to ensure exist:
- DDD layer labels for the layers involved
- Bounded context label for the feature's BC
- Aggregate labels for each aggregate in scope
- Type labels (`type::epic`, `type::story`)

**3b. Create Milestone**

Use `mcp__gitlab__create_milestone` with:
- Title: from FEATURE_SPEC.md Meta → Target Milestone (or derive from feature name)
- Description: from ISSUE_SPEC.md Overview section (first 2 paragraphs)

**3c. Create Epic Issues**

For each Epic in FEATURE_SPEC.md Section 8:
- Use `mcp__gitlab__create_issue` with the epic issue template from `references/gitlab-templates.md`
- Apply labels: `type::epic` + BC label + relevant DDD layer labels
- Assign to the milestone created in 3b

**3d. Create Story Issues**

For each Story within each Epic:
- Use `mcp__gitlab__create_issue` with the story issue template from `references/gitlab-templates.md`
- Apply labels: `type::story` + BC label + aggregate labels + DDD layer labels
- Assign to the same milestone
- Reference the parent epic issue in the description

**Important:** Always confirm with the user before creating GitLab artifacts. Show a summary of what will be created (N labels, 1 milestone, N epics, N stories) and wait for approval.

### Step 4 — GSD Delivery

If GSD is selected, follow this sequence. Read `references/gsd-mapping.md` for the full mapping rules.

**4a. Generate REQUIREMENTS.md**

Write `.planning/REQUIREMENTS.md` populated from FEATURE_SPEC.md:
- Each Epic → one requirement group
- Each Story → one requirement within the group
- Acceptance criteria from Section 4 → requirement acceptance criteria
- Open questions from Section 7 → "Prerequisites" section

**4b. Seed Phase Directories**

For each Epic in FEATURE_SPEC.md Section 8, create a phase context file:
- Create `.planning/phases/{NN}-{epic-slug}/{NN}-CONTEXT.md`
- Populate from FEATURE_SPEC.md Section 3 (aggregates) and Section 5 (integration contracts)
- Include relevant DDR constraints from Section 6

Do NOT create PLAN.md files — those are generated by `gsd:plan-phase` using the CONTEXT.md as input.

### Step 5 — Report Created Artifacts

After creation, report:

```
## Created Artifacts

### GitLab (if applicable)
- Milestone: {title} ({URL})
- Labels created: {count} ({list})
- Epic issues: {count}
  - {title} ({URL})
- Story issues: {count}
  - {title} ({URL})

### GSD (if applicable)
- REQUIREMENTS.md: .planning/REQUIREMENTS.md
- Phase contexts: {count}
  - .planning/phases/{NN}-{slug}/{NN}-CONTEXT.md
```

## Key Principles

- **Confirm before creating GitLab artifacts.** These are visible to the team. Always show a dry-run summary first.
- **Labels are additive.** Never delete existing labels. Only create missing ones.
- **GSD files are seeds, not finished plans.** CONTEXT.md provides the starting point for `gsd:plan-phase`. Do not over-specify.
- **Traceability is preserved.** Every GitLab issue references the DDD source docs. Every GSD context file references the aggregated spec.
- **Sprint decomposition follows rules.** Read `references/sprint-decomposition.md` for sizing and dependency ordering.
