# TPLAN- Node (Test Plan)

**Node type:** Test Plan
**Prefix:** `TPLAN-`
**Directory:** `/14_Outputs/testplans/`
**Ephemeral:** Yes (generated on demand; never manually versioned)

## When to Use

Test Plans are generated artifacts that derive test scenarios from linked FLOW- nodes. They compile the Shadow QA scenarios from each flow into an executable test suite. TPLANs are never manually authored; they are produced by the COMPILER during test preparation or release planning.

> **Key Rule:** TPLANs are ephemeral. Always regenerate when any linked FLOW's Shadow QA changes. Never edit TPLANs directly — edit the FLOW's `## Shadow QA` section instead.

---

## Quick Template (Generated Structure)

The COMPILER generates TPLAN files automatically. Expected structure:

```yaml
---
id: TPLAN-{FLOW-ID}-{N}  # N is sequence number if multiple flows
milestone: {M}
module: {Module}
wiki_snapshot_ref: "2025-01-15T10:30:00Z"  # Timestamp of wiki state at generation
linked_flows: ["[[FLOW-{ID}]]"]
status: active  # Always active; ephemeral nodes are not deprecated
---
```

```markdown
# TPLAN-{ID}

## Coverage Matrix

| Flow | Happy Path | Edge Cases | Fault Paths |
|------|------------|------------|-------------|
| [[FLOW-{ID}]] | ✅/❌ | {count} | {count} |

## Scenarios

### [[FLOW-{ID}]] — {Flow title}

**Happy Path:** Given [[ACT-{Actor}]] with {preconditions}, when [[CMD-{Command}]] fires, then {entity} transitions to `{state}` and {postcondition}.

**Edge Case:** Given {violated guard condition}, when [[CMD-{Command}]] executes, then {expected rejection or partial outcome}.

**Fault Path:** Given {system/integration failure}, then {rollback behavior} and {error manifestation}.

(Repeat for each linked flow)
```

## Frontmatter Fields

| Field | Required? | Rules | Example |
|-------|-----------|-------|---------|
| `id` | Yes | Must start with `TPLAN-` followed by flow ID and sequence | `TPLAN-OrderFulfillment-01` |
| `type` | Yes | Must be `test_plan` | `test_plan` |
| `milestone` | Yes | Current milestone identifier | `M1` |
| `module` | Yes | Must exist in `modules.md` | `sales` |
| `wiki_snapshot_ref` | Yes | ISO 8601 timestamp of wiki state at generation | `"2025-01-15T10:30:00Z"` |
| `linked_flows` | Yes | Array of FLOW- IDs (wikilinks) | `["[[FLOW-OrderFulfillment]]"]` |
| `status` | Yes | Always `active` for ephemeral nodes | `active` |

Note: `version` field is omitted for ephemeral nodes.

---

## Body Structure

### Required Sections

1. **`## Coverage Matrix`** — Table summarizing scenario coverage per linked flow. Indicates whether happy path, edge cases, and fault paths are present.
2. **`## Scenarios`** — One subsection per linked flow, containing:
   - Happy Path scenario
   - Edge Case scenario(s) as needed
   - Fault Path scenario(s) as needed

All scenarios must be sourced from the corresponding FLOW- node's `## Shadow QA` section. Copy them verbatim; do not rewrite.

---

## Schema Rules

- **Ephemeral Generation:** TPLANs are generated by the COMPILER, not manually authored. LINT rule: `ephemeral_edit_forbidden`.
- **Staleness Check:** If `wiki_snapshot_ref` predates the last modification timestamp of any linked FLOW- node, the TPLAN is stale and must be regenerated before a TRUN test run. LINT: `stale_test_plan`.
- **Scenario Completeness:** For each linked FLOW, the TPLAN must include at least the Happy Path scenario. Edge and Fault paths are mandatory if present in the FLOW's Shadow QA. LINT: `missing_happy_path`.
- **No Direct Edits:** Never edit TPLAN content directly. To change test scenarios, edit the source FLOW's Shadow QA section and regenerate.

---

## Related Nodes

- **FLOW**: Source of all test scenarios. FLOW's `## Shadow QA` section is the single source of truth.
- **TRUN**: Test Run node consumes TPLANs; TRUN references the TPLANs it executed.
- **FRS**: FLOWs are traced back to FRS; TPLAN inherits that traceability via FLOW.

---

## Notes

TPLANs exist solely to provide a versioned, snapshot test plan for a specific TRUN execution. They allow test results to be reproduced exactly even if wiki content changes later. The `wiki_snapshot_ref` provides traceability to the exact wiki state when the plan was generated.
