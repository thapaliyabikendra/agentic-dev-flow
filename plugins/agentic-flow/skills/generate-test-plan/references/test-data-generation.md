# Test Data Generation Rules

Used by Step 6.5 of `generate-test-plan` (Generate Test Data per TC) to produce concrete or templated values for every field a TC's Steps reference. The skill consults this file for the recipe that turns a field's constraints (from the data-model fact sheet) plus the TC's intent (happy-path / required-missing / max-length / duplicate / etc.) into a Test Data section entry.

This file does NOT generate values for itself. The skill walks the rules below per field, per TC; the values land in each TC's `## Test Data` section.

## Templated value vocabulary

The skill emits four placeholder tokens. The test-suite skill (phase 2) interpolates them at code-generation time. Until phase 2 ships, the tokens stay literal in the TC files.

| Token | Replaced at codegen with | Use when |
|---|---|---|
| `{timestamp}` | `Date.now()` (epoch ms) | Need a value unique per test run |
| `{counter}` | An incrementing per-spec-file counter | Need a value unique per test within a single spec |
| `{uuid}` | A generated UUID | Need a guaranteed-unique value, length doesn't matter |
| `{tcNumber}` | The TC's own number (e.g. `TC001`) | Need a value identifiable to a specific TC |

**Combination is allowed and encouraged.** `TC{tcNumber}-RT-{timestamp}` produces something like `TC001-RT-1746604800000` at runtime — short, unique, traceable back to the TC. This is the workhorse pattern for happy-path string fields.

**Order of preference:**
1. `{tcNumber}` + `{timestamp}` — most common; identifies the TC, guarantees uniqueness.
2. `{tcNumber}` alone — when timestamp uniqueness isn't needed (e.g. local fixtures cleaned between runs).
3. `{uuid}` — only when length doesn't matter and human-readability isn't valued.
4. Pure literal — for FRS-stated values like `Category: "Digital Transactions"`.

Avoid `{counter}` unless within-spec uniqueness is genuinely needed (it rarely is — `{timestamp}` is sufficient for cross-test isolation).

## Directive vocabulary (matrix-driven TCs)

For TCs generated by the Required Coverage Matrix (max-length, duplicate, required-missing, etc.), the Test Data entry uses a `human description → directive(args)` form. The directive is what the test-suite skill uses to validate or generate the value at codegen time.

| Directive | Args | Meaning |
|---|---|---|
| `violatesMaxLength(N)` | max length in chars | Generate a string exactly `N+1` characters long |
| `violatesMinLength(N)` | min length | Generate a string of length `N-1` (or empty if N=1) |
| `duplicate(value)` | the value to duplicate | Reference Pre-existing State — this value MUST already exist in DB |
| `invalidFormat(kind)` | `email \| url \| date \| phone \| number \| decimal` | Generate a value that violates the named format |
| `outOfRange(min, max)` | numeric bounds | Generate a number outside `[min, max]` (below min if min>0, above max otherwise) |
| `whitespaceOnly` | (none) | Generate `"   "` (3 spaces) |
| `empty` | (none) | Empty string `""` |
| `null` | (none) | Literal null (skip the fill step entirely; not the same as empty) |
| `injection` | `sql \| xss \| script` | Generate a known injection payload for the kind |

**Always emit BOTH the human description AND the directive.** The description is for the human reviewer; the directive is for the codegen. Example:

```
Template Name: 51-character string → violatesMaxLength(50)
Code: existing value 'TC-DUP-001' → duplicate('TC-DUP-001')
Email: 'not-an-email' → invalidFormat(email)
```

## Generation rules per field type × TC intent

### String fields (text, textarea, varchar)

Constraints captured in the data-model fact sheet: required, max length, min length, uniqueness, format (regex), whitespace-trimmed, etc.

| TC intent | Test Data entry |
|---|---|
| Happy path (valid input) | `Field Name: TC{tcNumber}-{kebab-feature}-{timestamp}` if max length ≥ 30, else `TC{tcNumber}{counter}` |
| Required missing | `Field Name: (empty) → empty` |
| Max length exceeded | `Field Name: <(maxLength+1)-character string> → violatesMaxLength({maxLength})` |
| Min length violation | `Field Name: <(minLength-1)-character string> → violatesMinLength({minLength})` |
| Duplicate value | `Field Name: <existing value> → duplicate('<existing value>')` (also requires Pre-existing State entry) |
| Whitespace-only | `Field Name: '   ' → whitespaceOnly` |
| Format invalid | `Field Name: <a value violating the format> → invalidFormat({kind})` |
| Special chars / injection | `Field Name: <payload> → injection(xss)` (or `sql`, `script`) |

**For happy-path uniqueness:** when the field is unique, the templated value MUST include a per-run uniqueness component — `{timestamp}` or `{uuid}`. `{tcNumber}` alone won't do because re-runs collide. The skill validates this: if a unique field's happy-path Test Data entry doesn't include `{timestamp}` or `{uuid}`, fail Step 6.5 verification.

**Field-name normalisation:** the Test Data entry uses the field's human-readable name as it appears in the FRS Form Fields section, NOT the camelCase formControl name. `Template Code` not `templateCode`. The test-suite skill's matching is case-insensitive substring, so step text *"Enter a value in the Template Code field"* matches the entry *"Template Code: ..."*.

### Numeric fields (integer, decimal)

Constraints: required, min, max, decimal precision.

| TC intent | Test Data entry |
|---|---|
| Happy path | `Field Name: <a value within [min, max]>` (e.g. midpoint, or a sensible default like 100) |
| Required missing | `Field Name: (empty) → empty` |
| Out of range (below min) | `Field Name: <min - 1> → outOfRange({min}, {max})` |
| Out of range (above max) | `Field Name: <max + 1> → outOfRange({min}, {max})` |
| Negative when not allowed | `Field Name: -1 → outOfRange(0, {max})` |
| Decimal precision exceeded | `Field Name: <value with too many decimal places> → invalidFormat(decimal)` |

If the FRS doesn't declare a min/max, the skill picks `100` for happy-path integers, `100.00` for decimals — flagged with a comment for human review.

### Boolean / toggle / checkbox

Constraints: default value, required.

| TC intent | Test Data entry |
|---|---|
| Happy path (set true) | `Field Name: true` |
| Happy path (set false) | `Field Name: false` |
| Toggle on→off (matrix) | `Field Name: false  (precondition: was true)` |
| Toggle off→on (matrix) | `Field Name: true  (precondition: was false)` |

### Select / dropdown / ng-select

Constraints: option list (if FRS enumerates options), required, multi-select.

| TC intent | Test Data entry |
|---|---|
| Happy path (option known from FRS) | `Field Name: '<option literal>'` |
| Happy path (option unknown) | `Field Name: (select first available option)` — flagged for human review |
| Required missing | `Field Name: (no selection) → empty` |
| Invalid option (rare) | `Field Name: <invalid string> → invalidFormat(option)` |

When the FRS enumerates options (e.g. `Category: { Digital Transactions, Retail, Corporate }`), the skill picks the first option as the happy-path value. When the FRS only describes the field type, the skill emits `(select first available option)` as a human-readable instruction — the test-suite skill emits a runtime "pick first option" pattern, no specific value baked in.

### Date / datetime

Constraints: required, min, max, format.

| TC intent | Test Data entry |
|---|---|
| Happy path (any date) | `Field Name: 2026-01-15` (a fixed sensible date — neither past-only nor future-only without FRS guidance) |
| Happy path (must be future) | `Field Name: <today + 30 days, ISO 8601>` — emit `{futureDate(30)}` token if the test-suite skill supports it; otherwise flag for human review |
| Past when must be future | `Field Name: 2020-01-01 → outOfRange(today, +infinity)` |
| Invalid format | `Field Name: 'not-a-date' → invalidFormat(date)` |

Date generation is the area I'm least confident about; if your FRSes routinely have date constraints (`must be ≥ today`, `must be within 30 days`), this rule may need extension with a `{futureDate(N)}` / `{pastDate(N)}` vocabulary in phase 2.

### File upload

Constraints: required, file types, max size.

| TC intent | Test Data entry |
|---|---|
| Happy path | `Field Name: <fixture: validUpload-<feature>.<ext>>` — references a fixture file the team provides; flagged for human review since the skill can't generate binary fixtures |
| Required missing | `Field Name: (no file) → empty` |
| Wrong file type | `Field Name: <fixture: wrongType.<ext>> → invalidFormat(fileType)` |
| Oversized | `Field Name: <fixture: oversized.<ext>> → violatesMaxLength({maxBytes})` |

File uploads always reference a fixture file. The skill emits the fixture name; the user's tests repo is responsible for providing the actual file. This is one of the few places where prose alone isn't enough — codegen needs an actual file path.

## Pre-existing State sub-section

For matrix-driven TCs that need database state (duplicate, FK delete, edit-non-existent, restore-after-soft-delete), the Test Data section gets a `### Pre-existing State` sub-block before `### Form Input`. Each entry is prose describing what the test-suite needs to set up:

```markdown
## Test Data

### Pre-existing State
- A Rule Template with Template Code "TC-DUP-001" already exists in the database (created via fixture or seed).
- The current user's Agent role has create permissions for Rule Templates.

### Form Input
* Template Code: TC-DUP-001 → duplicate('TC-DUP-001')
* Template Name: TC{tcNumber} Duplicate Test {timestamp}
* (...other fields with happy-path values...)
```

The Pre-existing State is prose only in phase 1. The test-suite skill emits these as TODO comments in the spec's `beforeEach` for the developer to wire to fixtures or seed scripts. When the team has a fixture library, phase 3 can add `<fixture: name>` syntax to make it executable.

## Generation rules per matrix-driven TC type

When a TC was generated by the Required Coverage Matrix in Step 6 Pass 2, Step 6.5 picks one or more *target fields* to violate, while keeping all other fields at happy-path values. The targeting rules:

| Matrix row | Target field selection | Other fields |
|---|---|---|
| Happy path | None violated | All happy-path |
| Required field missing | The specific required field named in the TC title | All others happy-path |
| Maximum length exceeded | The specific text field named in the TC title | All others happy-path |
| Duplicate value | The specific unique field named in the TC title; pre-existing state names the duplicate value | All others happy-path |
| Invalid format | The specific typed field named in the TC title | All others happy-path |
| Cross-field comparison violation | Two fields involved in the comparison rule, set to violating values; TC title identifies the rule | All others happy-path |
| Whitespace-only | The specific required field named in the TC title (whitespace-only value) | All others happy-path |
| Authorization | None (all fields happy-path); pre-existing state names the unauthorized user | All happy-path |
| Multi-tenancy isolation | None violated; pre-existing state names the cross-tenant record | All happy-path |
| Concurrent edit | None violated; pre-existing state names the stale-row scenario | All happy-path |

**One field violated per TC, by design.** The matrix produces one TC per violation; one TC violates one rule. Multi-violation TCs (e.g. "max length AND duplicate AT THE SAME TIME") are out of scope — they're not testing one rule, and they make failure analysis ambiguous.

## When generation fails

Some fields will resist generation:
- The FRS captures the field but no constraints (no length, no format, no type beyond "string") — happy-path generation can produce something reasonable, but max-length / format-violation TCs can't be generated; emit the TC with a TODO directive.
- The FRS captures the field with a constraint the directive vocabulary doesn't cover (e.g. *"Field must be a comma-separated list of department codes"*) — emit the literal happy-path constraint as prose, no directive.
- The field is dynamic (`subject identifier` populated from another field's selection) — emit `(populated by step N)` as the value, no directive. The test-suite skill knows to skip the fill step.

A TODO Test Data entry looks like:

```
* Field Name: TODO — constraint not parseable; manual fill needed
```

Same honest-failure model as TODO selectors. Better than fabricating a value the test would type and then fail on.

## Verification rules (Step 6.5 self-checks)

After generating Test Data for a TC, verify:

1. Every field referenced by a `fill` / `select` / `toggle` step has a Test Data entry — or the entry is explicitly marked TODO.
2. Every TC with a unique field has the field's happy-path value containing `{timestamp}` or `{uuid}`.
3. Matrix-driven TCs have exactly one field with a directive (one violation per TC).
4. Pre-existing State entries reference values that match the directive arguments (e.g. `duplicate('TC-DUP-001')` → state describes a record with Code `TC-DUP-001`).
5. No two TCs in the same use case have the same happy-path values for unique fields without `{timestamp}` (would collide on parallel runs).

Failures here block Step 9 (emission). Fix or flag with TODO before writing the file.
